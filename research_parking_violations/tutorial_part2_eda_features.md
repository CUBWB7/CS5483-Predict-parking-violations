## 第一部分：探索性数据分析（EDA）

### 1. 什么是 EDA，为什么要做

**EDA（Exploratory Data Analysis，探索性数据分析）** 是数据挖掘流程中最基础、也最容易被低估的一步。

打个比方：假设你刚搬进一间新厨房，冰箱里放了一堆你从没见过的食材。你会怎么做？你一定会先把每样东西拿出来看看：这是什么？新不新鲜？有没有过期的？哪些味道可以搭配？哪些食材数量充足、哪些只有一点点？——EDA 做的就是同样的事情，只是对象从食材变成了数据。

具体来说，EDA 要回答以下几类问题：

- **数据长什么样？** 每个特征的分布是什么形状？有没有极端值？是不是均匀分布还是集中在某个范围？
- **数据有没有缺失？** 哪些列有空值？缺了多少？缺失的原因是什么？该怎么填补？
- **特征之间有没有关系？** 哪些特征和目标变量（我们要预测的东西）相关性强？特征之间有没有冗余（两个特征基本在说同一件事）？
- **有没有异常模式？** 数据里有没有不符合常理的情况？有没有需要特殊处理的子群？

**记住一点：EDA 不是目的本身，而是为后面的特征工程和建模做准备。** 你在 EDA 中发现的每一个规律、每一个异常，都会直接指导你后面怎么构造特征、怎么选模型、怎么处理噪声。如果跳过 EDA 直接建模，就像蒙着眼睛做菜——也许能做出来，但大概率不好吃。

### 2. 我们数据的关键发现

我们的数据集有大约 **607 万行**，**10 个特征**，目标变量是 `invalid_ratio`（违规率，取值在 0 到 1 之间）。评估指标是 **Spearman 相关系数**——它衡量的是预测值和真实值的"排名一致性"，也就是说，你不需要预测出精确的数值，只需要把违规率从高到低的排序排对就行（后续教程会详细解释这个指标）。

下面是 EDA 中最重要的几个发现：

#### 2.1 目标变量的 U 型分布

如果你画出 `invalid_ratio` 的分布图（直方图），你会看到一个非常独特的 **U 型**：

- **15.86%** 的样本违规率 = 0（完全没有违规）
- **26.74%** 的样本违规率 = 1（全部违规）
- 其余样本分布在 0 到 1 之间，但占比明显更少

这看起来很不寻常——为什么这么多极端值？正常来说，违规率不应该大部分在中间附近吗？

答案藏在一个关键特征里：`total_count`。

想想看：如果一个区域在某个时段只检查了 **1 辆车**，那违规率只可能是两个值——0（这辆车没违规）或者 1（这辆车违规了），数学上不可能出现 0.5 这样的中间值。就好比你只考了一道判断题，成绩要么 0 分要么 100 分，永远不可能是 63 分。只有当检查的车多了（比如 10 辆里违规 3 辆），才会出现 0.3 这样的中间值。

这就是 U 型分布的根源：**大量小样本（total_count 很低）产生了大量 0 和 1 的极端值**。

#### 2.2 total_count 与噪声的关系（最重要的发现！）

`total_count`（检查总数）是我们数据中**最强的预测因子**，它和违规率的 Spearman 相关系数达到了 **-0.297**——这意味着检查数量越多的区域-时段，违规率越低、也越稳定。

我们按 `total_count` 分组来看统计数据：

| total_count 范围 | 样本数 | 平均违规率 | 标准差 |
|---|---|---|---|
| 1 | 126,289 | 0.664 | 0.472 |
| 2-3 | 94,785 | 0.582 | 0.382 |
| 4-10 | 144,112 | 0.459 | 0.296 |
| 11-30 | 110,449 | 0.354 | 0.213 |
| 31+ | 24,365 | 0.300 | 0.170 |

这张表揭示了一个核心问题：**小样本噪声**（Small Sample Noise）。

当 `total_count = 1` 时，标准差高达 0.472——接近最大可能值（0.5）的极限。这说明数据波动极其剧烈，一个样本的违规率完全是"随机"的。而当 `total_count >= 31` 时，标准差降到了 0.170，数据就稳定多了。

**生活类比**：你去一家新餐厅，只吃了 1 次，恰好那次厨师状态不好，你就在网上给差评——这公平吗？评价可能完全不准确。但如果你连续去了 30 次，综合评价就非常靠谱了。`total_count` 就是你"去餐厅的次数"，次数越多，对这家店的评估就越可靠。

这个发现对我们的建模有两个重要启示：
- 对于 `total_count` 大的样本，模型可以比较自信地预测
- 对于 `total_count` 小的样本，模型需要更"保守"，不能太相信极端值——后续建模中我们可能会给不同 total_count 的样本不同的权重

#### 2.3 空间模式

我们的数据有两个空间特征：`longitude_scaled`（经度，已归一化到 0-1）和 `latitude_scaled`（纬度，已归一化到 0-1）。如果单独看它们和违规率的相关系数，数值并不高。

但如果你把所有数据点画在一张平面图上——x 轴是经度、y 轴是纬度，用颜色深浅表示违规率——你会发现一个清晰的图景：**不同区域的违规模式差异非常大**。

有的区域一直是"重灾区"（违规率持续偏高，可能是商业繁忙区、停车位紧张的地方），有的区域则一直比较"守规矩"（可能是住宅区、或停车管理严格的区域）。这说明空间位置本身蕴含着很强的信息——只是原始的浮点数经纬度不是模型能直接利用的好形式。这正是后面我们要用空间特征工程来解决的问题。

Paper 4（Liu & Chen 2025）的研究也支持了这一点：他们发现道路密度、人口密度等建成环境因素是停车违规的主要驱动因素，而且这种影响在不同区域、不同时间段存在显著差异（即"时空异质性"）。

#### 2.4 时间模式

我们有三个时间特征：

- **月份（month_of_year，取值 1-12）**：和违规率的 Spearman 相关系数为 **-0.091**，存在季节性波动。通常夏季违规率较高（出行需求旺盛），冬季较低（天气冷出行少，或假期因素）。
- **星期几（day_of_week，取值 1-6）**：工作日和周末的停车行为模式明显不同。工作日的办公区违规率更高，而周末商业娱乐区的违规率可能升高。
- **小时（hour，取值 6-19）**：这个范围正好覆盖了停车检查的工作时间。午间时段（12-14 点）通常违规率更高——可能因为午餐时间停车需求剧增但车位紧张。Paper 3（Gao 2019）还发现了有趣的时间自相关现象：违规行为在约 3 小时后有一个明显的"回波"，说明违规模式存在时间上的滞后效应。

#### 2.5 天气特征：弱相关但别急着扔掉

数据中有 4 个天气特征：
- `Precipitations`（降水量）
- `HauteurNeige`（积雪高度）
- `Temperature`（温度）
- `ForceVent`（风力）

如果你计算它们各自和违规率的 Spearman 相关系数，数值都很低——几乎看不到什么关系。你可能会想：那这些天气变量是不是没用？该不该直接扔掉？

**先别急！弱相关不等于没用。**

**类比**：单独问"今天下不下雨"可能对一个人的迟到概率影响不大（坐地铁的人不受影响）。单独问"住在哪个区"也不够（天气好的时候开车很顺畅）。但如果你问"住在郊区 + 下大雨 + 周一早高峰"，这个组合条件下迟到的概率就非常高了。

同样的道理，"下雨"这个条件在某些区域、某些时段可能会显著改变违规率。Paper 5（Sui 2025）通过 SHAP 分析（一种解释模型决策的方法）发现：天气因子中**湿度**最显著，而且天气和时间之间存在**非线性交互**——也就是说，天气的影响不是简单的"线性叠加"，而是和具体情境有复杂的交互关系。这告诉我们：天气特征的价值需要通过"交叉特征"来释放，这是后面特征工程的内容。

#### 2.6 缺失值处理

好消息是我们的数据非常"干净"，缺失值极少：

- **HauteurNeige（积雪高度）**：约 **2.7%** 缺失。这个特征本质上接近一个二元变量——绝大多数时候积雪为 0（没有雪），只有冬天少数时段才有积雪。所以缺失值用 **0 填充**是合理的（合理假设：没有记录 = 没有下雪）。
- **ForceVent（风力）**：仅约 **0.1%** 缺失，数量极少。用 **中位数填充** 即可。为什么用中位数而不是均值？因为中位数对极端值更"免疫"——如果有一天刮了台风导致风力数值特别大，均值会被拉高，但中位数不会受影响。

```python
# 缺失值填充示例
train_df['HauteurNeige'].fillna(0, inplace=True)
median_wind = train_df['ForceVent'].median()
train_df['ForceVent'].fillna(median_wind, inplace=True)
```

### 3. EDA 常用可视化方法

做 EDA 时，"画图"远比"看数字"有用。以下是几种最常用的可视化方法：

1. **各特征的直方图（Histogram）**：看每个特征的分布形状。是正态分布（钟形曲线）？还是偏态分布（往一边倾斜）？有没有离群的异常值？

2. **目标变量的分布图**：在我们的项目中，就是画 `invalid_ratio` 的直方图。你会清楚地看到那个 U 型分布——这立刻就告诉你"数据不是正态分布的，需要特殊处理"。

3. **特征相关性热力图（Correlation Heatmap）**：用颜色矩阵一目了然地展示所有特征两两之间的相关系数。颜色越深（红或蓝）表示相关性越强，接近白色表示几乎无关。

4. **空间散点图（Spatial Scatter Plot）**：以经度为 x 轴、纬度为 y 轴，每个点的颜色深浅表示该位置的违规率。这可以直观地揭示"哪些地理区域违规严重"。

5. **时间维度折线图（Time Series Line Plot）**：画出按小时 / 星期几 / 月份分组的平均违规率折线图，可以一眼看出时间规律。

```python
# 示例 1：目标变量分布图
import matplotlib.pyplot as plt

plt.figure(figsize=(10, 6))
plt.hist(train_df['invalid_ratio'], bins=50, edgecolor='black', alpha=0.7)
plt.xlabel('违规率 (invalid_ratio)')
plt.ylabel('样本数量')
plt.title('目标变量分布 —— 注意两端的 U 型')
plt.show()
```

```python
# 示例 2：Spearman 相关性热力图
import seaborn as sns

corr_matrix = train_df.corr(method='spearman')

plt.figure(figsize=(12, 8))
sns.heatmap(corr_matrix, annot=True, cmap='coolwarm', center=0, fmt='.3f')
plt.title('特征 Spearman 相关性热力图')
plt.tight_layout()
plt.show()
```

```python
# 示例 3：按小时分组的平均违规率折线图
hourly_mean = train_df.groupby('hour')['invalid_ratio'].mean()

plt.figure(figsize=(10, 5))
plt.plot(hourly_mean.index, hourly_mean.values, marker='o', linewidth=2)

```

## 第二部分：特征工程

### 4. 什么是特征工程

**特征工程（Feature Engineering）** 是指从原始数据中创造、组合、变换出更有信息量的新特征，让模型更容易学到数据中的规律。

**打个比方**：你手上有面粉、鸡蛋、糖——这些就是"原始数据"。如果你把面粉、鸡蛋、糖直接端上桌，谁也不知道怎么吃。但如果你先把面粉和鸡蛋混合搅拌、加糖调味、放进烤箱做成蛋糕——同样的原材料，却变成了一种更容易"消化"的形式。特征工程做的就是类似的事情：把原始数据"烘焙"成模型更容易理解和利用的形式。

**为什么特征工程这么重要？** 在表格数据（Tabular Data）的竞赛和实际应用中，有一句广为流传的话：

> **"特征工程决定了你的天花板，模型选择只决定你离天花板有多近。"**

换句话说，与其花大量时间尝试各种花哨的模型、精心调参数，不如先花时间把特征做好。好的特征能让一个简单模型跑出优秀的成绩，而差的特征即使用最先进的模型也无力回天。这在 Kaggle 等数据竞赛平台上已经被反复验证。

---

### 5. 空间特征工程（重点！）

空间信息是我们项目中最有价值、但也最需要精心加工的部分。EDA 告诉我们不同区域的违规模式差异很大，现在我们就来把这个发现转化为模型能用的特征。

#### 5.1 为什么原始经纬度不够好

我们的数据有 `longitude_scaled` 和 `latitude_scaled` 两个特征，都是 0 到 1 之间的浮点数。直接把这两个数丢给模型，有什么问题？

**核心问题：模型很难从两个浮点数中"认出"这是哪个街区。**

想象一下：有两个位置，经度分别是 0.5000 和 0.5001。作为人类，你根本不知道这两个点是不是在同一条街上、属不属于同一个街区。模型面临同样的困难——它不知道 0.5000 和 0.5001 应该被看作"同一个区域"还是"不同区域"。而且，经度和纬度是连续值，城市中的违规模式却是按区域"块状分布"的（整个商业区违规率高，整个住宅区违规率低）。连续值和块状模式之间存在天然的"不匹配"。

我们需要把连续的坐标变成离散的**"区域标签"**，让模型能明确地知道"这些点都属于同一片区域，它们有相似的违规模式"。

#### 5.2 坐标网格化（Grid Encoding）

**原理**：把整个城市像棋盘一样划分成均匀的小方格。每个方格就是一个"区域"，每个数据点根据它的坐标被分配到某个方格中。

**方法**：对经度和纬度分别除以网格大小（grid size），然后取整数部分。

```python
# 网格化示例
grid_size = 0.001  # 由于坐标已归一化到 0-1，0.001 大致对应城市中较小的块

# 把连续坐标变成网格编号
train_df['grid_lon'] = (train_df['longitude_scaled'] / grid_size).astype(int)
train_df['grid_lat'] = (train_df['latitude_scaled'] / grid_size).astype(int)

# 组合成唯一的区域 ID
# 比如经度网格 500、纬度网格 730 → 区域 ID "500_730"
train_df['grid_id'] = train_df['grid_lon'].astype(str) + '_' + train_df['grid_lat'].astype(str)
```

**举例**：假设某个停车点的经度是 0.500、纬度是 0.730：
- 经度网格 = int(0.500 / 0.001) = 500
- 纬度网格 = int(0.730 / 0.001) = 730
- 区域 ID = "500_730"

同一个方格内的所有点都会得到相同的区域 ID，模型就知道它们属于"同一片区域"了。

**网格大小的选择很关键**：
- **太小**（比如 0.0001）：每个格子里样本太少，统计不稳定——类似 total_count=1 的问题
- **太大**（比如 0.01）：把差异很大的区域硬塞到同一格里，丢失了空间细节
- **建议**：从 0.001 左右开始尝试。更好的做法是同时创建多个尺度的网格（比如 0.001 细网格 + 0.005 粗网格），让模型既能看到局部细节，也能看到大范围的区域趋势

#### 5.3 区域聚类（Spatial Clustering）

除了规规矩矩的"棋盘格"，还可以用聚类算法让数据自己"告诉"我们哪些点应该算同一个区域。

- **KMeans**：预先指定要分成 K 个区域，算法根据点与点之间的距离自动把数据分组。优点是简单快速，缺点是你得自己决定 K 是多少（可以用"肘部法则"来辅助选择）。
- **DBSCAN**：不需要预先指定数量，它根据数据点的密度自动发现"扎堆"的区域。密度高的地方形成一个簇（Cluster），密度低的孤立点被标记为噪声。

**网格化 vs 聚类的区别**：网格化是"一刀切"的均匀切分，不管某个区域有 1 万个数据点还是只有 3 个点，格子大小都一样。聚类则会根据数据点的实际分布来划分——热闹的市中心会被分成更多更细的区域，偏远郊区的几个点可能合并成一个大区域。

```python
from sklearn.cluster import KMeans

# 用 KMeans 把空间分成 50 个区域
coords = train_df[['longitude_scaled', 'latitude_scaled']].values
kmeans = KMeans(n_clusters=50, random_state=42, n_init=10)
train_df['spatial_cluster'] = kmeans.fit_predict(coords)
```

#### 5.4 区域统计特征（Target Encoding 思路）

有了区域 ID 之后，最强大的操作来了：**计算每个区域的历史平均违规率**，作为新特征加到每条数据上。

直觉很简单：如果某个区域过去一直违规率很高（比如商业繁忙区、停车位紧张），那未来这个区域的违规率大概率也高。反之亦然。这本质上是一种 **Target Encoding（目标编码）**——用目标变量（invalid_ratio）的统计值来编码分类特征（区域 ID）。

```python
# 计算每个区域的平均违规率
region_mean = train_df.groupby('grid_id')['invalid_ratio'].mean()
train_df['region_avg_violation'] = train_df['grid_id'].map(region_mean)
```

看起来很简单对吧？**但是，这里隐藏着一个非常危险的陷阱——数据泄露（Data Leakage）！**

##### 什么是数据泄露？为什么 Target Encoding 容易泄露？

**类比**：想象你在参加考试。做第 5 题的时候，你偷偷翻了一下答案册，看到了第 5 题的标准答案——然后你"参考"答案写下了自己的回答。考场上你当然能做对，但这完全不代表你真的学会了。换一套卷子你就暴露了。

Target Encoding 的泄露问题一模一样：当你计算区域 A 的平均违规率时，如果你把当前这条数据自身的 `invalid_ratio` 值也算进去了，模型就等于"偷看了自己的答案"。训练阶段分数会虚高（因为模型记住了答案），但面对从未见过的新数据时，表现就会严重下降——这就是过拟合（Overfitting）。

##### 如何防止泄露？方法一：K-Fold Target Encoding

把训练集分成 K 折（比如 5 折）。对第 1 折中的数据，用第 2-5 折的数据来计算区域均值；对第 2 折的数据，用第 1、3、4、5 折的数据来计算。以此类推。这样**每条数据在计算自己的编码值时，绝对不会用到自己的目标值**。

```python
from sklearn.model_selection import KFold
import numpy as np

# K-Fold Target Encoding 示例
kf = KFold(n_splits=5, shuffle=True, random_state=42)
train_df['region_target_enc'] = np.nan

for train_idx, val_idx in kf.split(train_df):
    # 用"其他折"的数据计算区域均值
    region_mean = train_df.iloc[train_idx].groupby('grid_id')['invalid_ratio'].mean()
    # 赋值给"当前折"
    train_df.loc[train_df.index[val_idx], 'region_target_enc'] = \
        train_df.iloc[val_idx]['grid_id'].map(region_mean)

# 对测试集：用全部训练集的区域均值来编码（因为测试集不参与训练）
test_region_mean = train_df.groupby('grid_id')['invalid_ratio'].mean()
test_df['region_target_enc'] = test_df['grid_id'].map(test_region_mean)
```

##### 如何防止泄露？方法二：加平滑（Smoothing）

即使做了 K-Fold，如果某个区域只有 3 个样本，那 3 个数据的均值也很不稳定。**平滑（Smoothing）** 的思路是：把区域均值"拉"向全局均值，样本越少的区域拉得越多。

```python
# 平滑公式：smoothed_mean = (n × region_mean + m × global_mean) / (n + m)
# n = 该区域样本数，m = 平滑参数（通常取 10-100）

global_mean = train_df['invalid_ratio'].mean()
m = 30  # 平滑参数

region_stats = train_df.groupby('grid_id')['invalid_ratio'].agg(['mean', 'count'])
region_stats['smoothed'] = (
    (region_stats['count'] * region_stats['mean'] + m * global_mean)
    / (region_stats['count'] + m)
)
```

平滑参数 `m` 的含义很直观：它相当于"你至少需要 m 个样本，才会比较相信这个区域自己的均值"。如果某个区域只有 5 个样本而 m=30，那它的编码值会非常接近全局均值——这是合理的，因为 5 个样本的信息量不足以做出可靠结论。

**最佳实践：同时使用 K-Fold + 平滑**，双重保险。

#### 5.5 论文中的空间特征思路

学术论文中有一些更精细的空间特征构造方法，可以给我们启发：

- **Paper 1（Vo 2025）**：使用了 19 个 **PoI（Point of Interest，兴趣点）** 距离特征——每个位置到最近的银行、医院、学校、超市等的距离。这些距离间接反映了区域的功能属性（商业区 vs 住宅区 vs 学校周边），不同功能区的违规模式自然不同。我们虽然没有 PoI 数据，但通过 Target Encoding 可以在一定程度上捕获类似的信息——因为区域的功能属性会反映在历史违规率中。
- **Paper 3（Gao 2019）**：发现 POI 密度（零售店密度、餐饮密度、医疗设施密度）是重要的空间特征，而且在不同的空间尺度上效果不同——近距离的 POI 密度和远距离的 POI 密度传递的信息不一样。这启发我们可以**同时使用粗细两种网格**（多尺度空间特征），让模型从不同"放大倍率"看空间格局。

### 6. 时间特征工程

#### 6.1 周期编码（Cyclical Encoding）

我们的时间特征有 `hour`（6-19）、`day_of_week`（1-6）、`month_of_year`（1-12）。直接把这些数字喂给模型有什么问题？

**核心问题：数值上的"距离"不等于现实中的"距离"。**

12 月（数值 12）和 1 月（数值 1）在数轴上相差 11，但在现实中它们是紧挨着的——12 月 31 日的下一天就是 1 月 1 日！如果你直接用原始数值，模型会认为 12 月和 1 月相差极远，但实际上它们的天气、行为模式可能非常相似。同理，星期六（6）和星期天之间也存在类似问题。

**解决方案**：用 **sin** 和 **cos** 函数把时间"画"到一个圆上。圆上的点天然就是首尾相连的——12 月和 1 月在圆上紧紧相邻！

```python
import numpy as np

# 小时的周期编码（24 小时为完整周期）
train_df['hour_sin'] = np.sin(2 * np.pi * train_df['hour'] / 24)
train_df['hour_cos'] = np.cos(2 * np.pi * train_df['hour'] / 24)

# 星期几的周期编码（7 天为完整周期）
train_df['dow_sin'] = np.sin(2 * np.pi * train_df['day_of_week'] / 7)
train_df['dow_cos'] = np.cos(2 * np.pi * train_df['day_of_week'] / 7)

# 月份的周期编码（12 个月为完整周期）
train_df['month_sin'] = np.sin(2 * np.pi * (train_df['month_of_year'] - 1) / 12)
train_df['month_cos'] = np.cos(2 * np.pi * (train_df['month_of_year'] - 1) / 12)
```

**为什么同时需要 sin 和 cos？** 想象一个时钟：sin 值告诉你"在时钟的上半还是下半"，cos 值告诉你"在时钟的左半还是右半"。只用 sin 的话，3 点和 9 点的 sin 值都是 0，模型分不清。加上 cos 之后，(sin, cos) 这对坐标可以唯一确定圆上的每一个位置，模型就不会混淆了。

Paper 1（Vo 2025）也使用了同样的技术：`sin(2pi x w/7)` 编码星期，`sin(2pi x (m-1)/12)` 编码月份。

#### 6.2 时段分箱（Time Period Binning）

除了精确的周期编码，还可以从"业务含义"的角度把小时划分为粗粒度的"时段"：

| 时段 | 对应小时 | 停车行为特点 |
|---|---|---|
| 早高峰 | 7-9 | 上班停车，匆忙中容易违规 |
| 上午 | 9-12 | 办公时间，停车相对稳定 |
| 午休 | 12-14 | 外出吃饭，短时停车多，违规率可能升高 |
| 下午 | 14-17 | 办公时间，类似上午 |
| 傍晚 | 17-19 | 下班高峰，停车位周转加快 |

```python
def get_time_period(hour):
    """把小时转换为有业务含义的时段"""
    if hour < 7:
        return 'early_morning'   # 清晨（数据中较少）
    elif hour < 9:
        return 'morning_rush'    # 早高峰
    elif hour < 12:
        return 'morning'         # 上午
    elif hour < 14:
        return 'lunch'           # 午休
    elif hour < 17:
        return 'afternoon'       # 下午
    else:
        return 'evening'         # 傍晚

train_df['time_period'] = train_df['hour'].apply(get_time_period)
```

**周期编码 vs 时段分箱，到底用哪个？** 答案是：**两个都用！** 它们提供的信息是互补的。周期编码保留了精细的时间信息（模型知道 10 点和 11 点是不同的），时段分箱提供了粗粒度的业务语义（模型知道"午休时段"是一种特殊的停车模式）。好的特征工程就是要从多个角度描述同一个事物。

### 7. 交叉特征

#### 7.1 什么是交叉特征

**交叉特征（Interaction Feature / Cross Feature）** 是指将两个或多个特征"组合"在一起，产生一个新特征。

**比喻**：你想预测一个人会不会迟到上班。单独知道"今天下雨"用处不大（有人坐地铁根本不受影响）。单独知道"住在郊区"也不够确定（天气好时开车半小时就到了）。但"住在郊区 + 下大雨 + 周一早高峰"的组合就非常有信息量——这个人很可能堵在路上迟到。每个条件单独都不够强，但组合起来就产生了"1+1>2"的效果。

在我们的项目中也是一样。前面 EDA 发现天气特征单独看几乎没用，但这不代表"下雨 + 午休时段 + 商业区"这个组合条件下的违规率没有规律。**交叉特征就是帮模型提前把这些"组合条件"准备好。**

#### 7.2 常用的交叉特征组合

以下是几类在我们项目中最有价值的交叉特征：

**1. 区域 x 时段**（预期价值最高）

不同区域在不同时段的违规模式可能完全不同。比如：商业区午间违规率高（午餐短时停车），住宅区早间违规率高（上班前抢车位）。

```python
# 创建区域×时段的组合特征
train_df['region_period'] = train_df['grid_id'] + '_' + train_df['time_period']

# 然后对这个组合 ID 做 Target Encoding（同样要用 K-Fold 防泄露！）
```

**2. 区域 x 星期几**

有些区域工作日和周末的差别特别大。比如 CBD 办公区工作日停车爆满、周末空空荡荡。

**3. 天气 x 时段**

"下雨 + 早高峰"和"下雨 + 下午"的影响可能完全不同——早高峰下雨，大家可能更着急、更容易违规停车。

**4. total_count 分箱 x 区域**

检查量大的区域和检查量小的区域，违规率模式不同。可以先把 total_count 分成几档（1, 2-3, 4-10, 11-30, 31+），再和区域组合——这样模型能区分"这个区域检查量大时的表现"和"检查量小时的表现"。

### 8. 天气特征增强

虽然原始天气特征单独和违规率相关性不高，但通过简单的变换可以挖掘出更多信息：

#### 8.1 二元化（Binarization）

把连续的天气数值简化为更直接的"是/否"判断——很多时候，"有没有下雨"比"下了多少毫米的雨"更有意义：

```python
# 是否下雨（降水量 > 0）
train_df['is_raining'] = (train_df['Precipitations'] > 0).astype(int)

# 是否有积雪
train_df['has_snow'] = (train_df['HauteurNeige'] > 0).astype(int)

# 是否极端温度（低于 5°C 或高于 30°C）
train_df['extreme_temp'] = (
    (train_df['Temperature'] < 5) | (train_df['Temperature'] > 30)
).astype(int)

# 是否大风（风力超过某个阈值，需要根据数据分布确定）
wind_threshold = train_df['ForceVent'].quantile(0.9)  # 取前 10% 作为"大风"
train_df['strong_wind'] = (train_df['ForceVent'] > wind_threshold).astype(int)
```

#### 8.2 天气窗口平均（Rolling Weather）

Paper 1（Vo 2025）使用了 **6 小时窗口的平均天气值**。为什么？因为天气对停车行为的影响有**滞后效应**——刚下完雨路面还是湿的、刚停完雪路面还是滑的，人们的出行决策会受到"过去几小时的天气"而非"此刻的天气"的影响。

虽然我们的数据集中每行是独立的（区域-时段组合），没有显式的时间序列顺序，但如果能获取同一天同一区域不同时段的数据，就可以计算"当日截至当前的平均天气"来近似这种滞后效应：

```python
# 如果数据中有日期信息（或者可以从月份+星期推断），可以做同日天气均值
# 这是一个简化的示例思路
daily_weather = train_df.groupby(['grid_id', 'day_of_week', 'month_of_year'])[
    ['Temperature', 'Precipitations']
].transform('mean')

train_df['daily_avg_temp'] = daily_weather['Temperature']
train_df['daily_avg_precip'] = daily_weather['Precipitations']
```

#### 8.3 Paper 5 的发现补充

Paper 5（Sui 2025）通过 **SHAP 分析**（SHapley Additive exPlanations——一种基于博弈论来解释模型每个特征贡献的方法）发现：天气因子中**湿度**是最显著的影响因素。虽然我们的数据没有直接的湿度特征，但降水量和积雪在一定程度上是湿度的代理变量。另外，SHAP 还揭示了天气与时间存在**非线性交互**——这意味着天气的影响不是"温度每升高 1 度，违规率就固定增加 X"这么简单，而是在不同条件下天气的影响方向和幅度都不同。这进一步验证了交叉特征的必要性。

### 9. 小结：特征工程的优先级

不同类型的特征，预期的"性价比"不同。下面这张表可以帮你合理安排时间：

| 特征类别 | 具体操作 | 预期重要性 | 实现难度 | 优先级 |
|---|---|---|---|---|
| **区域统计特征** | 网格化 + K-Fold Target Encoding | ★★★★★ | 中等（注意防泄露） | **最高** |
| **total_count 变换** | 对数变换 log(1+x)、分箱 | ★★★★☆ | 低 | **高** |
| **时间周期编码** | sin/cos 编码小时、星期、月份 | ★★★☆☆ | 低 | **高** |
| **交叉特征** | 区域 x 时段、区域 x 星期 | ★★★★☆ | 中等 | **高** |
| **时段分箱** | 按业务含义划分早高峰/午休等 | ★★★☆☆ | 低 | 中 |
| **天气二元化** | 是否下雨 / 下雪 / 极端温度 | ★★☆☆☆ | 低 | 中 |
| **空间聚类** | KMeans / DBSCAN 聚类 | ★★★☆☆ | 中等 | 中 |
| **天气窗口 / 交互** | 6h 平均天气、天气 x 时段 | ★★☆☆☆ | 中等 | 低 |

**建议的推进顺序**：
1. **先做区域统计特征和 total_count 变换**——这两个投入产出比最高，能带来最大的提升
2. **再做时间特征（周期编码 + 时段分箱）和交叉特征**——锦上添花，进一步捕获细节
3. **最后做天气增强**——边际改善，但实现简单，顺手加上不亏

## 学习要点回顾

这篇教程涉及了几个关键概念，简要总结一下：

1. **EDA 是数据挖掘的"侦察兵"**：不要急着建模，先把数据摸透。你在 EDA 中发现的每个规律和异常，都会直接指导特征工程的方向。

2. **小样本噪声是我们项目的核心挑战**：`total_count` 小的样本数据不可靠，这导致了目标变量的 U 型分布，也影响了模型的学习效果。理解这一点是理解整个项目的钥匙。

3. **特征工程 > 模型选择**：在表格数据上，好的特征比花哨的模型更重要。把时间花在特征工程上，回报最大。

4. **空间特征需要离散化**：原始经纬度浮点数不好用，需要通过网格化或聚类变成"区域"，再做统计编码。

5. **Target Encoding 必须防泄露**：用 K-Fold + 平滑双重保险来避免模型"偷看答案"。这是初学者最容易犯的错误之一，后果也最严重。

6. **周期特征用 sin/cos 编码**：让模型理解 12 月和 1 月是相邻的，而不是数轴上距离为 11 的两个点。

7. **交叉特征捕捉组合效应**：单个特征弱相关不代表没用。多个弱特征组合后可能产生强信号——这就是"1+1>2"的思想。
